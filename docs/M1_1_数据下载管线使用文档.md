# M1.1 数据下载管线使用文档

本文说明 QuantLab 当前整理到 `quantlab.data` 包里的数据下载接口，重点说明腾讯 5 分钟接口、Baostock 日线接口的区别，以及腾讯接口在 5 分钟数据补齐中的正确用法。

## 1. 总体原则

当前数据下载接口按“先下载到隔离 staging，再验证，再写回正式 Qlib 数据目录”的方式组织。

不要让原始下载脚本直接覆盖正式数据目录。正式目录包括：

```text
quantlab/01_data_daily/A_data        # 日线 Qlib provider
quantlab/02_data_5min/min_data       # 5 分钟 Qlib provider
```

下载接口只负责生成原始 CSV 和 `_download_summary.csv`。是否写回正式 provider，要由后续验证脚本或 wrapper 决定。

## 2. 当前接口一览

| 数据源 | 当前用途 | 包内入口 | 兼容脚本入口 | 输出位置 |
| --- | --- | --- | --- | --- |
| 腾讯 Finance | 5 分钟 fallback 下载 | `python -m quantlab.data.tencent_5min_download` | `scripts/download_tencent_5min_range.py` | 用户指定的 staging 目录 |
| Baostock | 日线 OHLCV 下载 | `python -m quantlab.data.baostock_daily_download` | `scripts/download_baostock_ohlcv_range.py` | 用户指定的 staging 目录 |

腾讯接口现在只作为 5 分钟数据的应急补源；Baostock 日线接口用于日线数据 staging。两者都不会直接写回 Qlib 正式数据目录。

## 3. 腾讯接口和 Baostock 接口的区别

### 3.1 数据频率不同

腾讯接口当前拉取的是 `m5` 5 分钟 K 线，URL 模式为：

```text
http://ifzq.gtimg.cn/appstock/app/kline/mkline?param={code},m5,,{count}
```

Baostock 日线接口调用的是 Baostock 的 `query_history_k_data_plus`：

```text
frequency="d"
adjustflag="2"
fields="date,open,high,low,close,volume,amount"
```

所以不要把腾讯接口当成日线主数据源，也不要把 Baostock 日线接口当成 5 分钟补源。

### 3.2 代码格式不同

系统内部统一使用 Qlib 风格代码：

```text
SH600000
SZ000001
```

下载器内部会自动转换：

| Qlib 代码 | 腾讯代码 | Baostock 代码 |
| --- | --- | --- |
| `SH600000` | `sh600000` | `sh.600000` |
| `SZ000001` | `sz000001` | `sz.000001` |

不支持的前缀会记录为 `bad_symbol`，不会中断全量下载。

### 3.3 volume 和 amount 处理不同

腾讯 `m5` 返回的 volume 单位是“手”。下载器会转换成“股”：

```text
volume = volume_hands * 100
```

腾讯接口当前没有直接使用独立成交额字段，amount 使用近似值：

```text
amount = close * volume
```

因此腾讯 5 分钟数据适合作为应急 fallback 和特征补齐来源，但 amount 不能视为和主数据源完全等价。

Baostock 日线接口直接使用 Baostock 返回的 `volume` 和 `amount`。

### 3.4 运行依赖不同

腾讯接口不需要 Baostock 登录，直接 HTTP 请求腾讯接口。它适合在 Baostock 5 分钟数据滞后、空行过多、或者需要快速验证某个日期 5 分钟数据可用性时使用。

Baostock 日线接口需要 Baostock 登录。日线接口在 `--resume` 模式下会先检查 staging 里已有的 CSV；如果目标股票都已有文件，会直接生成 summary，不会再登录 Baostock。

## 4. 腾讯 5 分钟接口怎么用

### 4.1 单独下载到 staging

最直接的用法是调用包内入口：

```bash
/home/gpu/.conda/envs/asrlab/bin/python -m quantlab.data.tencent_5min_download \
  --data-dir quantlab/01_data_daily/A_data \
  --market all \
  --start 2026-06-02 \
  --end 2026-06-02 \
  --staging-dir quantlab/08_archive_tmp/tencent_raw/tencent_5min_20260602 \
  --recent-bars 640 \
  --workers 1 \
  --resume
```

等价的兼容脚本入口：

```bash
/home/gpu/.conda/envs/asrlab/bin/python scripts/download_tencent_5min_range.py \
  --data-dir quantlab/01_data_daily/A_data \
  --market all \
  --start 2026-06-02 \
  --end 2026-06-02 \
  --staging-dir quantlab/08_archive_tmp/tencent_raw/tencent_5min_20260602 \
  --recent-bars 640 \
  --workers 1 \
  --resume
```

`--data-dir` 这里只用于读取股票池文件：

```text
quantlab/01_data_daily/A_data/instruments/all.txt
```

腾讯下载结果不会写入这个目录。

### 4.2 常用参数

| 参数 | 说明 |
| --- | --- |
| `--data-dir` | 用来读取 `instruments/<market>.txt` 的 Qlib provider |
| `--market` | 股票池，通常用 `all` |
| `--symbols-file` | 指定股票列表文件；适合 smoke test 或定向修复 |
| `--start` / `--end` | 从腾讯 recent bars 中保留的日期范围 |
| `--staging-dir` | 输出目录，必须显式指定 |
| `--recent-bars` | 每个股票向腾讯请求的最近 5 分钟 bar 数 |
| `--workers` | 并发数；全市场保守使用 `1` |
| `--resume` | 已有非空 CSV 时跳过，便于断点续跑 |
| `--limit` | 限制股票数量，适合小样本验证 |
| `--status-every` | 每处理多少只股票写一次 summary |

### 4.3 输出文件

每只成功下载的股票会生成一个 CSV：

```text
<staging-dir>/SH600000.csv
```

列格式为：

```text
date,symbol,open,high,low,close,volume,amount,adjustflag
```

其中：

- `date` 是 5 分钟 bar 时间，格式为 `YYYY-MM-DD HH:MM:SS`
- `symbol` 是 Qlib 风格代码，例如 `SH600000`
- `volume` 已从“手”换算成“股”
- `amount` 是 `close * volume` 的近似值
- `adjustflag` 当前固定为 `3`

下载器还会生成汇总文件：

```text
<staging-dir>/_download_summary.csv
```

列格式为：

```text
symbol,code,status,rows
```

常见状态：

| status | 含义 |
| --- | --- |
| `ok` | 本次成功下载并写出 CSV |
| `skipped_existing` | `--resume` 下已有非空 CSV，本次跳过 |
| `empty` | 腾讯返回中没有落在 `--start` 到 `--end` 范围内的 5 分钟 bar |
| `bad_symbol` | 不支持的股票代码前缀 |
| `error:<reason>` | 请求或解析失败 |
| `exception:<type>:<reason>` | worker 中出现未捕获异常 |

做完整性检查时，`ok` 和 `skipped_existing` 都应计入可用数据，不能只统计 `ok`。

### 4.4 小样本 smoke test

如果只是确认腾讯接口是否可用，可以先跑几只股票：

```bash
/home/gpu/.conda/envs/asrlab/bin/python -m quantlab.data.tencent_5min_download \
  --data-dir quantlab/01_data_daily/A_data \
  --market all \
  --start 2026-06-02 \
  --end 2026-06-02 \
  --staging-dir /home/tmp/tencent_5min_smoke_20260602 \
  --limit 20 \
  --workers 1 \
  --resume
```

检查 summary：

```bash
python - <<'PY'
import pandas as pd
path = "/home/tmp/tencent_5min_smoke_20260602/_download_summary.csv"
df = pd.read_csv(path)
print(df["status"].value_counts())
print(df.head())
PY
```

如果 `ok + skipped_existing` 比例较高，再扩大到全市场。

## 5. 腾讯 fallback 完整更新流程

单独的 `quantlab.data.tencent_5min_download` 只负责下载 staging。如果目标是“更新 5 分钟正式 provider，并继续跑 Kronos”，应使用 wrapper：

```bash
/home/gpu/.conda/envs/asrlab/bin/python -u scripts/run_tencent_5min_update_and_kronos.py \
  --end 2026-06-02 \
  --gpu 0
```

这个 wrapper 会执行完整流程：

1. 读取当前 5 分钟 provider 日历：

```text
quantlab/02_data_5min/min_data/calendars/5min.txt
```

2. 读取当前 Kronos 5 分钟 manifest：

```text
quantlab/04_kronos_5min/2026/pipeline_5min_kronos_2026_gpu0_20260527_145447/manifest.csv
```

3. 推断缺失日期范围。
4. 调用 `python -m quantlab.data.tencent_5min_download` 下载到隔离 staging。
5. 校验必需指数：

```text
SH000300
SH000903
SH000905
```

6. 校验全市场可用比例，统计口径为 `ok + skipped_existing`。
7. 通过 5 分钟 Qlib 更新脚本把 staging 写回正式 5 分钟 provider。
8. 跑缺失日期的 Kronos 5 分钟推理。
9. 在 `run_logs/kronos_5min_pipeline/` 下写入本次运行的 markdown 记录和 JSON 状态。

所以：

- 排查腾讯源是否可用：用 `python -m quantlab.data.tencent_5min_download`
- 真正补齐 5 分钟 provider 并继续 Kronos：用 `scripts/run_tencent_5min_update_and_kronos.py`

## 6. Baostock 日线接口怎么用

Baostock 日线接口用于日线 OHLCV staging：

```bash
/home/gpu/.conda/envs/asrlab/bin/python -m quantlab.data.baostock_daily_download \
  --data-dir quantlab/01_data_daily/A_data \
  --market all \
  --start 2026-06-01 \
  --end 2026-06-05 \
  --staging-dir quantlab/08_archive_tmp/baostock_daily_raw/baostock_daily_20260601_20260605 \
  --resume
```

兼容脚本入口：

```bash
/home/gpu/.conda/envs/asrlab/bin/python scripts/download_baostock_ohlcv_range.py \
  --data-dir quantlab/01_data_daily/A_data \
  --market all \
  --start 2026-06-01 \
  --end 2026-06-05 \
  --staging-dir quantlab/08_archive_tmp/baostock_daily_raw/baostock_daily_20260601_20260605 \
  --resume
```

每只股票输出：

```text
<staging-dir>/SH600000.csv
```

列格式：

```text
symbol,date,open,high,low,close,volume,amount
```

Baostock 日线 staging 验证通过后，再用现有 Qlib dump/update 管线写回日线 provider。例如在适用场景下使用：

```text
scripts/dump_baostock_daily_to_qlib.py
```

具体是否写回、写回哪些日期、是否需要备份，应按当次修复或更新任务决定。

## 7. 运行建议

1. 原始下载永远先落到 staging。
2. staging 目录名带上数据源、频率和日期范围。
3. 保留 `_download_summary.csv`，不要只保留单票 CSV。
4. 腾讯 5 分钟 fallback 写回前必须检查指数和全市场可用比例。
5. resume 续跑时，统计可用数据要用 `ok + skipped_existing`。
6. 腾讯 5 分钟 amount 是近似值，涉及成交额敏感的任务要单独标注数据源。
7. 正式数据目录写回后，要再检查 calendar、manifest 或 Qlib 构建日志，不能只看下载脚本退出码。
